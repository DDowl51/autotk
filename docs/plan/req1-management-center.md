# 需求 1：管理中心（后端 + Electron 桌面应用）

> 已按用户反馈改为 **Electron 桌面应用 + 本地后端 + 文件夹工作流**。

## 目标

- 手机启动后能在**管理中心看到**（设备列表、在线、运行统计、报错告警）。
- **1.1** 管理中心**批量改设置**，推送到一台/多台/全部，**即时生效**。
- **1.2** **文件夹工作流发视频**：操作员电脑上建一个根文件夹，下面若干**子文件夹（名字=设备名）**，子文件夹里的视频=该设备**当天要发**的视频；管理中心自动把视频发到对应手机的 TikTok，**发过的做记录、不重复发**。

## 为什么 Electron（而非网页）

文件夹工作流要**读本地磁盘**（监视根文件夹、按设备名分发）。网页做不到自由访问本地文件，Electron 可以。

## 架构（云 Hub + Electron + 局域网优先/中转降级）

手机**不一定**和操作员 PC 同局域网 → 需要一个 **云协调中心 Hub**（always-on，国内可访问）让两边都连上；视频**同局域网直传、跨网云中转降级**。

```
                云 Hub（always-on，国内可访问）
        设备协调 / 状态 / 配置 / 任务路由 (+ 视频中转，降级时)
          ▲                                   ▲
   WS(状态/指令)│                               │ WS
   ┌──────────┴───────────┐          ┌─────────┴──────────────┐
   操作员电脑 Electron 应用            手机 1..N（autotk，几百台）
   看板/批量设置/发布进度              连 Hub；配置即时生效；
   文件夹监视 + 本地 LAN 文件服务        收发视频任务、上报状态
```

- **手机和 Electron 都连云 Hub**（穿 NAT、跨网都行）；配置/状态/任务等小数据走 Hub。
- **视频传输两路**：
  - **同局域网（优先）**：Hub 牵线，Electron 上报自己的 LAN 地址，手机探测可达 → **直接从 Electron 局域网 HTTP 下载**（快、省带宽）。
  - **跨网（降级）**：手机够不到 Electron 局域网 → Electron 把视频上传 Hub → 手机从 Hub 下载。
- **Hub 与 #5 共用账号/部署**（同一云、同一套账号）。

## 设备命名与文件夹（自动创建）

- 管理软件**首次启动弹窗**，让操作员指定**根文件夹位置**（存设置里）。
- **每台手机首次连上来 → 自动在根文件夹下创建 `<设备名>/` 子文件夹**（设备名来自手机上报：iOS 设备名或 autotk 里设的别名）。操作员把当天视频丢进对应子文件夹即可，**无需手动建/映射**。
- 手机用 `deviceId`（同 #5）唯一标识；设备名是给人看的别名 + 文件夹名。

## 1.2 发视频：调度、去重、文案

### 调度（按你的反馈）

- 每台设备：读它子文件夹里**当天**的 N 条视频。
- 把 N 条**平均分布到当天的运行时间段**（沿用该设备的 `taskWindows`）。
- 间隔**不等距**：在平均间隔上加随机抖动（避免机械等距）。
- 也支持**显式定时**（给某条视频指定时间）作为可选。

### 去重记录（发过不再发）

- Electron 侧维护**清单**（SQLite / json）：键=设备+文件名(+大小/修改时间)，记已发时间。**不动用户的原文件**。
- 可选：发完把文件**移动/复制到 `已发送/` 子目录**，让操作员肉眼可见（做成开关）。

### 文案（你没定，我给建议）

推荐**多来源、按优先级取**，灵活又简单：

1. **每视频旁挂同名 `.txt`**：`clip1.mp4` + `clip1.txt`（里面就是该视频文案）——最精确，操作员想精确控每条就用这个。
2. 子文件夹放 **`captions.txt`**（一行一条）：没有同名 txt 时，从这里**随机/轮流**取一条。
3. 都没有 → 用**文件名**（去扩展名）当文案，或留空。
4. 末尾可统一追加**话题标签**（设备级配置，如 `#fyp #foryou`）。

- 也支持 #2 的占位符（`{emoji}` 等）增加变体、降风控。

封面、谁可看：默认，不调整。

## 1.1 批量改设置

- 配置就是现有 `AutomationParams`（+ 固定回复 #2、评论匹配词 #3 等）。抽到 `packages/shared` 让 手机/Electron 共用类型与 `validateParams`。
- UI：选若干设备 → 编辑配置或套模板 → 下发 → 手机**即时生效**（运行中热替换当前参数，下个批次用新值）。

## 告警

- 手机端脱困失败/被弹窗困住/长时间无进展 → 上报 Electron 看板高亮告警（#4 里"告警管理中心"对应这里）。

## 分阶段执行

- **阶段 1**：Electron 骨架 + 内置 WS/HTTP 后端 + SQLite；手机端连上报状态；看板（在线/运行/统计/告警）。
- **阶段 2**：批量设置（1.1）；shared 抽 schema；即时生效。
- **阶段 3**：文件夹工作流发视频（1.2）：监视+调度+去重+局域网传+TikTok 发布自动化（真机调，依赖 #4）+ 进度回报。

## 依赖 / 顺序

- 账号体系与 #5 共用（已确认同一套）。
- 1.2 强依赖 **#4**（发布流程极易遇权限/引导弹窗）。
- 放最后；阶段 1 可较早起（给 #4 告警、#5 用量提供通道）。

## 风险 / 注意

- 几百台手机的 WS 连接 + 局域网视频服务都压在操作员 PC 上 → 后端要扛得住（连接管理、并发下载限流）。
- 同质视频批量发风控高 → 每台不同文案 + 抖动间隔 + 符合运行时段（已设计）。
- TikTok 发布流程随版本/机型变 → 坐标可维护、多机型。
- 手机自动发视频需**相册写权限**（#4 策略里相册允许对应此处）。

## 开放问题（剩余）

- **机房拓扑**：几百台手机和操作员 PC 是否同一局域网？（决定走"Electron 内置后端+局域网直传" 还是 "独立云后端中转"。强烈影响视频传输设计。）
  A: 不是同一局域网， 但最好也做一下统一局域网下的传输方式， 如果不在同一局域网则降级中转
- 设备命名：操作员在看板里命名 + 文件夹名匹配，可接受吗？（还是手机端设置里设名字？）
  A: 这样比较麻烦， 最好是每一台设备新连上来之后， 自动创建一个对应设备名的文件夹， （文件夹的创建位置在设置中指定， 管理软件初次启动的时候弹窗提示）
- 发布失败重试策略 / 当天没发完的视频结转明天？
  A： 发布失败做告警
